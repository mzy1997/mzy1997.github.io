使用方法：最常用的方法，首先划分为有效等价类、无效等价类，即有效输入、无效输入，再对两个子集做进一步划分

注意事项：等价类划分虽然是最常使用的设计方法之一，但不可无脑用之，因为挑选出的测试数据尽为该集合中有限的几个，在某些场景，测试数据需要穷举，则等价类划分不适用

2. 边界值

解释：针对输入、输出边界值设计用例

使用方法：1 < n < 10，测试数据：小于最小值、大于最大值，等于最大、最小值，边界内。进一步划分，可以加入大于最小值一个单位长度，小于最大值一个单位长度

注意事项：

1. 当存在多个边界条件，需要对边界值进行组合时，如有变量X、Y
    1. 强边界：每个测试用例包含多个边界条件，input(Xmin, Ymax)等等
    2. 弱边界：每个测试用例只包含一个边界条件，input(xnormal, ymax)等等
    3. 全边界：整个测试用例包含所有边界条件，即强边界与弱边界的父集
2. 不仅仅可针对输入域，也可以针对输出域

3. 场景法

解释：针对业务场景进行划分，划分标准为该业务场景的若干流程分支

使用方法：根据场景的若干流程，划分出基本流与备选流，基本流可理解为我们通常所讲的“主流程”。

实例：

采购申请发起、审批业务场景

- 基本流：
    - 发起采购申请、审批通过
- 备选流：
    - 发起采购申请、审批驳回
    - 发起采购申请、返回用户修改参数、驳回
    - 发起采购申请、请他人审批、审批通过
    - 发起采购申请、请他人审批、驳回

4. 错误推断法

解释：测试人员根据直觉或者积累，推断程序可能出现的错误

使用方法：

针对不同的测试对象，进行不同的推断，针对性设计用例

实例：

1. 测试输入框
    1. 输入为空
    2. 输入超长字符
    3. 输入null
    4. 输入特殊字符
    5. 输入转义字符
    6. 输入通配符
    7. 输入html标签
    8. 输入前面带空格的字符串
    9. 输入后面带字符的字符串
    
    ...........
    

注意事项：该方法完全靠测试人员的日常积累

5. 正交实验法

解释 ：正交，线性代数中意为两向量点乘为0，即互相垂直，也可译为相互独立，不可替代。正交实验法，在测试工作中，将多个可组合的测试因素及其水平归纳到对应的正交表中

使用方法：

1. 根据需求形成因素状态表
2. 确定所采用的正交表
3. 将正交表中的数字、字母由文字代替
4. 取一行为一个测试用例

实例：

有3个测试因素，分别为字体、字号、颜色，这些因素对应的水平为，字体：宋体、黑体、仿宋；字号：10、20、30；颜色：黑色、蓝色、红色

1. 根据需求形成因素状态表
    
    
    | 字体 | 字号 | 颜色 |
    | --- | --- | --- |
    | 宋体 | 10 | 黑色 |
    | 黑体 | 20 | 蓝色 |
    | 仿宋 | 30 | 红色 |
2. 确定采用的正交表
    
    正交表公式：$Ln(m^k)$
    
    参数解读：L —正交表代号，n —实验次数、行数、用例数，m—最大水平数，k—因素个数
    
    最终我们采用的正交表为3因素、3水平，也就是$L9(3^3)$正交表（网上找正交表，如果因素、水平较少也可以自己画笛卡尔积表，然后去除非独立数据，去除标准为：1.每列中每个水平出现的次数相同；2.任何水平的组合只出现一次）
    
3. 将正交表中的数字、字母由文字代替
    
    如宋体==1，黑体==2，仿宋==3
    
4. 取一行为一个测试用例

6. 判定表

解释：对于存在的条件进行判定，得出将要做出的行动，并将这一过程记录在表中

判定表的主要元素：条件桩、动作桩（条件的集合、动作的集合）

                                 条件项、动作项（单个条件、单个动作）

使用方法：

1. 识别条件和动作
2. 生成判定表
3. 简化判定表

实例：

场景：当前合同限额为5w，如果大于等于5w，且选择的采购方式需要电子签，则走电子签章；如果低于5w，则不会发起合同；如果大于等于5w，但所选的采购方式未配置电子签，则走普通签章；

1. 识别条件和动作
    1. 条件：是否高于5w、采购方式是否需要电子签
    2. 动作：不签合同、电子签、普通签
2. 生成判定表
    
    
    | 条件桩 | 条件项 | 条件项 | 条件项 | 条件项 |
    | --- | --- | --- | --- | --- |
    | 是否高于5w | 1 | 1 | 0 | 0 |
    | 是否需要电子签 | 0 | 1 | 1 | 0 |
    | 动作桩 | 动作项 | 动作项 | 动作项 | 动作项 |
    | 不签合同 |  |  | x | x |
    | 电子签 |  | x |  |  |
    | 普通签 | x |  |  |  |
3. 简化判定表
    
    因为当合同金额低于5w时，不会发起合同，因此不存在签章的动作（第三条件项、第四条件项），因此可以进行简化
    
    当是否高于5w为0，无论是否电子签的值为什么，都不签合同
    
    | 条件桩 | 条件项 | 条件项 | 条件项 |
    | --- | --- | --- | --- |
    | 是否高于5w | 1 | 1 | 0 |
    | 是否需要电子签 | 0 | 1 | - |
    | 动作桩 | 动作项 | 动作项 | 动作项 |
    | 不签合同 |  |  | x |
    | 电子签 |  | x |  |
    | 普通签 | x |  |  |

7. 因果图

解释：人都是复杂的，由人引发的各种事件同样是复杂的，一个事件的走向往往受多个关联或不关联的因素影响，因果图就是通过梳理一个事件的成因及其最终走向而形成。

方法：

1. 找到事件的所有影响因素，以及尽可能多的结果
2. 梳理各因素之间的关系、因素与结果之间的关系
3.  画出因果图，转换为决策表
4. 根据决策表设计测试用例
