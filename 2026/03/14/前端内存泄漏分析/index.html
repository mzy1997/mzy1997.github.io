什么是内存泄漏

  1. js中的内存生命周期

  2. js的垃圾回收

    js的内存管理是自动的，每当我们创建函数、对象的时候就会自动分配内存，相应的，如果一个函数、对象无人引用，或者从全局对象为起点，无法递归访问到时，就会自动执行垃圾回收

  3. 引起内存泄漏的几种情况

    1. 意外的全局变量

    2. 未及时清理的定时器或回调函数

    3. 错误的使用闭包

    4. 未移除事件监听

    5. DOM元素引用没有被清除

3. Memory

三种方式来记录内存情况：

  • Heap snapshot(堆快照)

  • Allocation instrumentation on timeline

  • Allocation sampling

  • Heap snapshot(堆快照)，

    • Summary，摘要视图，按照构造函数对对象进行分组
    • Comparison, 比较视图，对比两个快照的对象变化情况
    • Containment, 包含视图，从全局变量出发，一层一层的往下探索
    • Statistics, 统计视图

页面中的几个关键词：

  • Constructor：构造函数。
  • Distance：（根）距离，对象与 GC 根之间的最短距离。
  • Shallow Size：浅层大小，单位：Bytes（字节）。
  • Retained Size：保留大小，单位：Bytes（字节）。
  • Retainers：持有者，也就是直接引用目标对象的变量。

  • Allocation instrumentation on timeline(随时间线展示JS内存分配情况)

    时间轴中主要的 3 种线：

    • 细横线：内存分配大小刻度线
    • 蓝色竖线：表示内存在对应时刻被分配，最后仍然活跃
    • 灰色竖线：表示内存在对应时刻被分配，但最后被回收

  • Allocation sampling（分配采样）

    通过分配采样我们可以很直观地看到代码中的每个函数所分配的内存大小

    分配采样结果页有 3 种视图可选:

    • Chart：图表视图
    • Heavy (Bottom Up)：扁平视图（调用层级自下而上，反向的函数调用过程）
    • Tree (Top Down)：树状视图（调用层级自上而下，正向的函数调用过程）

排查思路

  1. 内存泄漏对于用户而言最直观的表现就是，访问速度变慢、或者多次访问后浏览器崩溃并提示OutOfMemory。后者就代表多次访问的页面存在内存泄漏，而让访问速度变慢的情况有很多，包括渲染或网络传输等方面的瓶颈

  2. 要排查是否因内存泄漏引起访问速度变慢，可以先使用TimeLine来观察操作相关页面时的内存占用变化情况，如果TimeLine上的JS Heap曲线呈现锯齿状，或短时间内上升幅度较小的阶梯状，就属于正常表现。如果曲线持续上升，且停止操作后较长时间内未下降回原值，就可能存在泄漏

  3. 通过TimeLine观察到页面可能存在泄漏后，可以继续使用Memory板块中的Heap snapshot来进一步定位，通过观察Summary（内存概要）、比较操作前、操作后的堆快招（Comparison），来进一步定位

注意事项

在分析内存泄漏之前，应当关闭浏览器所有的拓展应用，避免非必要因素对结果产生影响
